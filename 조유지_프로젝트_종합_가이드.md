## 🛠️ 당신이 구현할 기능들

### **In Scope** (당신이 해야 할 것)
- ✅ **덱(Deck) CRUD**: 덱 이름, 설명, 공개 여부 관리
- ✅ **카드(Card) CRUD**: 앞면/뒷면, 태그, 미디어 관리
- ✅ **SM-2 알고리즘**: 복습 난이도(rating)에 따라 다음 복습일 계산
- ✅ **복습 세션**: 사용자가 오늘 복습할 카드를 조회하고 평가
- ✅ **Kafka 이벤트 발행**:
  - `card.reviewed`: XP 적립 트리거 (engagement-owner로 전송)
  - `card.review.due`: 복습 리마인더 (platform-owner로 전송)
- ✅ **복습 통계 API**: 일별/주별 복습 수, 정답률 조회

### **Out of Scope** (다른 팀이 담당)
- ❌ AI 카드 자동 생성 (learning-ai-owner)
- ❌ 게이미피케이션 로직 (engagement-owner)
- ❌ 알림 발송 (platform-owner)

## 🏗️ 기술 아키텍처

### 핵심 도메인 모델

```
Deck (덱)
  ├─ id (UUID)
  ├─ userId (사용자 ID)
  ├─ title, description, isPublic
  └─ cards (1:N)
        │
        ├─ Card (카드)
        │   ├─ id
        │   ├─ front, back (학습 내용)
        │   ├─ nextReviewDate (다음 복습일)
        │   ├─ easeFactor (SM-2 난이도 지수, 초기값 2.5)
        │   ├─ interval (복습 간격, 일 단위)
        │   └─ repetition (반복 횟수)
        │
        └─ ReviewSession (복습 세션)
            ├─ id
            ├─ userId, deckId
            ├─ startedAt, completedAt
            ├─ cardCount
            └─ CardRating (카드별 평가)
                ├─ cardId
                ├─ rating (0:Again, 1:Hard, 2:Good, 3:Easy)
                └─ nextReviewDate (계산된 다음 복습일)
```

### SM-2 알고리즘

**SM-2는 flashcard 복습의 최적 간격을 계산하는 알고리즘입니다.**

```
난이도별 처리:
- Again (0): interval = 1, repetition = 0 리셋
- Hard (1): interval 유지, repetition += 1
- Good (2): interval = interval × EF, repetition += 1
- Easy (3): interval = interval × EF × 2, repetition += 1

EF (Ease Factor) 계산:
EF_new = EF_old + (0.1 - (5-rating) × (0.08 + (5-rating) × 0.02))
최솟값: 1.3
```

**예시**:
- 초기: interval=1, EF=2.5, repetition=0
- "Good" 평가 → interval = 1×2.5 = 2.5일, EF = 2.36
- 2.5일 후 다시 "Good" → interval = 2.5×2.36 ≈ 6일, EF = 2.22

## 📊 API 엔드포인트 (당신이 구현할 것)

### Deck API
```
POST   /api/v1/decks                    # 덱 생성
GET    /api/v1/decks                    # 내 덱 목록 (페이지네이션)
GET    /api/v1/decks/{deckId}           # 덱 상세 조회
PUT    /api/v1/decks/{deckId}           # 덱 수정
DELETE /api/v1/decks/{deckId}           # 덱 삭제 (soft delete)
```

### Card API
```
POST   /api/v1/decks/{deckId}/cards           # 카드 생성
GET    /api/v1/decks/{deckId}/cards           # 덱 내 카드 목록
GET    /api/v1/decks/{deckId}/cards/{cardId}  # 카드 상세
PUT    /api/v1/decks/{deckId}/cards/{cardId}  # 카드 수정
DELETE /api/v1/decks/{deckId}/cards/{cardId}  # 카드 삭제
```

### Review Session API
```
POST   /api/v1/review-sessions                           # 복습 세션 시작
GET    /api/v1/review-sessions/{sessionId}               # 세션 상태
POST   /api/v1/review-sessions/{sessionId}/cards/{cardId}/rate  # 카드 평가
GET    /api/v1/review-sessions/{sessionId}/cards/today   # 오늘 복습 카드 조회
```

### Statistics API
```
GET    /api/v1/stats/review-summary           # 일별/주별 복습 통계
GET    /api/v1/stats/streak                   # 연속 복습 일수
GET    /api/v1/stats/accuracy                 # 정답률
```

## 📚 참고 기술 스펙

### SM-2 알고리즘 (Spaced Repetition)
- **논문**: "Optimization of learning" (S.M. Wozniak, 1990)
- **용도**: Anki, Quizlet, RemNote 등 모든 주요 flashcard 앱이 사용
- **핵심 개념**: 
  - **Interval**: 카드를 다시 본 사이의 날짜 수
  - **Ease Factor**: 카드 난이도 지수 (1.3~3.0)
  - **Repetition**: 누적 반복 횟수

### Modulith (Spring Framework)
- Java 기반 **모듈화 아키텍처** 프레임워크
- `card`, `srs` 등 모듈을 독립적으로 개발하되, 명확한 인터페이스로 연결
- 모듈 간 순환 참조 방지, 테스트 격리 가능

### Kafka 이벤트
- **Event-Driven Architecture**: 서비스 간 느슨한 결합
- `card.reviewed`, `card.review.due` 같은 주제(topic)로 다른 서비스에 알림
- AVRO 스키마로 메시지 검증
